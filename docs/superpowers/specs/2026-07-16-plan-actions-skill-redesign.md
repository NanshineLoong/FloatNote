# 下一步 Skill 重设计

- 日期：2026-07-16
- 状态：已确认、实现并验证
- 范围：`src-tauri/resources/skills/plan-actions/SKILL.md`

## 目标

更新内置 `plan-actions` Skill，使它帮助用户从不确定或宽泛的目标中理清当前方向，并通过多轮、按需的对话形成一小批近期真正可执行的行动。Skill 不负责建立完整项目 backlog，也不在一次调用结束后持续跟踪用户。

中文 UI 元数据改为：

- 名称：`下一步`
- 简介：`理清方向，定下眼前真正能做的事`

稳定 Skill ID 仍为 `plan-actions`。用于模型发现的英文 `description` 只描述触发情境，不概括工作流；建议为：

> Use when the user needs to clarify an uncertain direction or decide a small set of near-term actions for a goal.

## 核心原则

采用自适应漏斗：先判断当前不确定性，再决定是否需要发散方向、追问或调研，最后收敛为极简方案和一批近期行动。只进行会改变方向、方案或行动的交互，不为遵循固定轮数而提问。

一次 Skill 调用只形成当前这一批行动。后续批次必须由用户在未来的显式调用或新会话中再次启动。

## 启动上下文

每次显式调用均视为一次新的规划过程，不依赖上次会话记忆：

1. 读取 `_tasks.md`，了解已有、进行中和可能重复的行动。
2. 读取用户明确指定的 Inbox、原方案或其他文档。
3. 不自动扫描全部 piece，也不自行猜测哪份文档是原方案。
4. 如果本次要推进的对象仍不清楚，只提出一个聚焦问题来确认。

## 自适应工作流

### 1. 定位

确认用户本次希望推进的可观察结果，并识别会显著影响方向的关键假设、约束或未知信息。

### 2. 发散

- 目标或方向模糊时，提出 2–4 个实质不同的前进方向。
- 每个方向只需说明核心收益、代价和适用条件，让用户选择、组合或补充。
- 用户已经明确方向时，简短复述目标和关键假设，跳过方向发散。
- 用户要求直接给出下一步时可以压缩讨论，但不能把未经确认的重大假设直接写成任务。

### 3. 收敛

- 一次只问一个会改变方向、方案或行动的关键问题。
- 外部事实可能改变选择时，助手可直接开始目标明确的调研，无需先征求许可。
- 调研前先形成具体验证问题；获得足以支持近期决策的信息后停止。
- 个人偏好、价值取舍和风险承受度由用户决定，不用外部资料替用户选择。
- 不妨碍开始的未知信息可转化为验证行动，而不是继续无限调研。

满足以下条件后进入行动设计：

- 本次结果足够明确；
- 当前方向已经选定；
- 关键约束和重大未知已澄清，或已转化为验证行动；
- 至少存在一个无需再次做重大选择即可开始的行动。

发现关键假设不成立时，可以退回定位、发散或收敛阶段。

### 4. 落地

1. 在聊天中一次展示完整的极简行动方案和近期行动。
2. 等待用户修改、删除、排序或确认，不在确认前写入。
3. 确认后默认新建独立的行动方案 piece。
4. 文档成功写入后，再询问是否将用户确认的非重复行动追加到 `_tasks.md`。
5. 用户可以明确要求跳过方案文档并直接写入 `_tasks.md`；这是对默认流程的显式覆盖。
6. 简短说明保存位置和写入结果，然后结束本次 Skill 调用。

## 极简行动方案

默认新建独立 piece，文件名从目标生成，例如 `发布个人网站行动方案.md`。不覆盖重名文档；重名时让用户选择新名称或明确指定更新已有文档。

```markdown
# <目标名称>行动方案

## 目标
一句话说明希望达成的可观察结果。

## 当前方向
说明这次选择怎样推进，以及一两条必要理由。

## 关键节点
- 只列出理解整体路径所需的少量阶段或里程碑。

## 近期行动
1. 结果可验证、通常可在一次专注工作中完成的行动。

## 决策依据
- 只保留会帮助未来继续规划的关键结论；使用外部资料时附来源链接。
```

结构规则：

- `目标`、`当前方向`、`近期行动`必需。
- `关键节点`仅在目标包含多个阶段时出现。
- `决策依据`仅在提问或调研实质影响方案，且未来继续规划会需要这些信息时出现。
- 不记录完整思考过程或冗长调研报告。
- 文档中的行动使用普通有序列表；只有 `_tasks.md` 使用 Markdown checkbox，避免形成两个任务状态来源。

## 近期行动契约

- 目标数量为 3–7 项；只有 1–2 项真正有价值时不为凑数增加任务。
- 每项只包含一个主要结果，以明确动词开头。
- 写明必要的对象、范围和产出，使任务脱离当前聊天仍能理解。
- 完成状态必须可判断，通常能在一次专注工作中完成。
- “研究”“处理”“推进”“优化”等动词必须补充可观察的产出。
- 按建议执行顺序排列。
- 允许短依赖链，但前置行动必须在同一批中且顺序清楚。
- 依赖长期外部结果、且本批没有安排前置工作的事项不加入。
- 不生成完整远期 backlog。

## 调研规则

- 只在外部事实可能改变方向或近期行动时调研。
- 区分资料事实、助手推断和仍未确认的信息。
- 重要资料保留来源链接。
- 信息冲突、不可得或仍不确定时不伪造结论；说明限制，并在必要时把验证转化为近期行动。

## 写入与安全边界

- 默认先写独立方案文档，再询问是否写入 `_tasks.md`。
- 写入任何文件前说明将写入什么以及写到哪里。
- 写入 `_tasks.md` 前重新读取现有内容，并进行语义去重；措辞不同但结果相同也视为重复。
- 只追加用户确认的行动，不删除、改写或重排已有任务。
- 不修改、清理或标记 Inbox 等源材料。
- 用户拒绝写入后停止，不重复等价请求，也不换工具绕过拒绝。
- 如果方案文档写入失败，报告原因，不假装已经保存；除非用户明确要求，否则不继续默认的任务写入步骤。

## Skill 形态

Skill 保持自包含且简洁，只更新 `SKILL.md`，不增加 scripts、references、assets 或额外说明文件。正文建议使用以下部分：

1. `Overview`
2. `Start With Context`
3. `Adaptive Workflow`
4. `Action Contract`
5. `Plan Document and Tasks`
6. `Research and Boundaries`

用正向流程和结构契约塑造输出；禁止性规则只用于写入权限、源材料保护和调用边界。正文保持英文，与现有内置 Skills 一致；中文只放在 FloatNote UI 元数据中。

## 验证设计

按照 `writing-skills` 的 RED–GREEN–REFACTOR 方法，在修改 Skill 前后使用相同的现实场景。基线验证不得向测试代理泄露预期修复。

1. **模糊目标**：应提供 2–4 个方向并等待选择，不能直接制造任务。
2. **明确目标**：应跳过机械发散，只处理真正阻碍行动的未知信息。
3. **必要调研**：应自主验证会改变决策的外部事实，保留来源，并在足够支持近期行动时停止。
4. **个人取舍**：应向用户提问，不能用搜索结果替用户决定价值偏好。
5. **近期范围**：只给当前一批行动，允许同批短依赖链，不生成完整项目 backlog。
6. **行动质量**：每项结果可验证、上下文完整、通常能在一次专注工作中完成；不为凑数添加任务。
7. **默认落盘**：先在聊天中确认完整方案，再创建独立行动方案 piece，之后才询问是否追加任务。
8. **显式覆盖**：用户明确要求时允许跳过方案文档，直接请求写入 `_tasks.md`。
9. **去重与保护**：不得添加语义重复任务，不得修改已有任务或源材料。
10. **调用边界**：完成本批行动后结束，不声称持续跟踪，也不自动生成下一批。
11. **失败处理**：写入拒绝或失败后应如实停止，不绕过确认或假装已经保存。

完成行为验证后，检查 frontmatter、名称规则和词数，并运行与内置 Skill 解析及目录展示相关的仓库测试。
