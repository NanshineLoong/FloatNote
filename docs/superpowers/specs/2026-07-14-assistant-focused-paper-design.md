# AI 助手聚焦纸张设计

## 背景

AI 助手的紧凑输入器在内容达到高度上限后可以放大。当前放大态存在三个问题：

1. Floating 模式下只显示覆盖全窗的灰色遮罩，纸张位于遮罩之后，任何外部点击又会立即收起放大态。
2. 放大态没有随窗口连续改变尺寸。Floating 和 Inline 各自呈现一套近似固定的宽度。
3. 编辑器仍像嵌套在卡片里的输入框，不符合“直接在纸张上书写、发送按钮悬浮在右下角”的目标。

本设计把放大态统一为与助手停靠模式无关的“聚焦纸张”，同时保留当前单一 CodeMirror `EditorView` 的状态连续性。

## 根因

### Floating 模式的层级错误

`#assistant-region` 在 Floating 模式下创建了 `z-index: 20` 的层叠上下文。当前遮罩直接挂在 `body` 下，使用 `z-index: 40`；输入器虽然设置为 `z-index: 41`，但仍被限制在父级的 `z-index: 20` 内，无法绘制到遮罩之上。

结果是遮罩可见而纸张不可见。遮罩本身允许点击穿透，文档级捕获监听器再根据点击坐标判断是否收起，因此用户感受到“全屏变灰，点击任何位置都会退出”。

### Floating 模式的尺寸覆盖

Floating 的紧凑输入规则 `#app.mode-floating #assistant-region .assistant-input-wrap.open` 具有比放大态规则更高的选择器权重，把输入器宽度固定回约 `240px`。Inline 没有这条覆盖，所以两种模式表现为两套尺寸，而非同一个响应式表面。

### 视觉层级重复

当前放大态同时绘制外层容器边框、内层 CodeMirror 输入边框、圆角和操作按钮，使它看起来像“放大的输入框嵌在 Dialog 中”。这与 FloatNote 的轻量写作界面不一致。

## 目标

- Floating 和 Inline 使用完全相同的放大输入体验。
- 放大态随窗口宽高连续变化，并保留安全边距。
- 编辑器表现为无边框的纸张书写面，而不是放大的表单控件。
- 点击纸张外部不收起；只允许 `Escape`、右上角关闭按钮或成功提交收起。
- 保留输入文本、光标、选区、引用 chip、撤销/重做和 IME 状态。
- 发送失败时保留输入，避免用户重新输入。
- 键盘、屏幕阅读器、减少动态效果和 Windows/macOS 均有明确行为。

## 非目标

- 不改变紧凑输入器的展开条件或普通聊天布局。
- 不重做消息历史、机器人 dock、引用候选列表或权限气泡。
- 不创建第二个 CodeMirror 实例，也不在两个实例之间同步编辑状态。
- 不引入富文本工具栏、标题栏、快捷键提示或新的格式化能力。
- 不改变 Enter 发送、Shift+Enter 换行以及引用候选的既有语义。

## 选定方案：聚焦纸张

曾考虑三种方向：

- **聚焦纸张（选定）**：居中纸张配轻遮罩，适合长输入，视觉最安静。
- **底部展开纸**：从 dock 原位向上生长，连续性强，但长文本的可用高度较少。
- **全窗写作画布**：输入接管内容区，沉浸最强，但失去原文参照且模式切换明显。

聚焦纸张采用“纯净纸张”版本：不显示标题、助手身份、工具栏、底部分隔线或快捷键提示。永久可见的操作只有右上角关闭按钮和右下角发送按钮。

## 交互设计

### 打开

- 紧凑输入达到现有高度阈值后，放大按钮保持可用。
- 点击放大按钮后，同一输入器移动到居中的聚焦纸张。
- 当前文本、选择、滚动位置、引用和编辑历史保持不变。
- 动画完成后 CodeMirror 获得焦点，并保持原有光标位置。

### 保持打开

- 点击纸张正文、关闭按钮或发送按钮均正常接收事件。
- 点击背景遮罩不会收起，也不会把事件传递给底层文档。
- 焦点限制在聚焦层内；Tab 可以在编辑器、关闭按钮和发送按钮之间移动。
- `@` 和 `/` 候选仍锚定当前光标，并绘制在纸张及遮罩之上。

### 收起

以下操作收起聚焦纸张：

- 按 `Escape`；若引用候选打开，第一次 `Escape` 只关闭候选，下一次才收起纸张。
- 点击右上角关闭按钮。
- 有效内容被后端接受并成功开始请求。

点击背景、切换 Floating/Inline 或调整窗口大小均不收起。

收起后输入器回到原 dock 位置。手动关闭和 `Escape` 保留内容；成功提交会按既有发送语义清空内容。

## 视觉设计

### 聚焦层

- 聚焦层覆盖应用视口并阻止底层交互。
- 遮罩只做轻度降噪，不成为主要视觉元素；浅色和深色模式均通过语义 token 配置。
- 纸张无可见边框，使用 `--color-surface`、适度圆角和柔和阴影建立层级。
- 不添加传统 Dialog 标题栏或分区 chrome。

### 编辑面

- CodeMirror 根节点、滚动区和内容区与纸张共享同一背景。
- 放大态移除输入器边框和内层圆角，视觉上只剩一张连续纸面。
- 内容 padding 连续响应窗口尺寸，在小窗口保持紧凑，在大窗口提供舒展留白。
- 为右下角发送按钮预留内容安全区，长文本和引用 chip 不会被按钮遮挡。

### 操作按钮

- 关闭按钮是右上角低强调 ghost icon button，并保留全局 focus-visible 样式。
- 发送按钮固定悬浮在纸张右下角，使用现有 primary button 语义和发送图标。
- 空内容时不把发送按钮切换为历史入口；聚焦态下空提交不收起，历史入口仍属于紧凑 dock。

### 响应式几何

- 纸张宽度：`min(920px, calc(100vw - 32px))`。
- 纸张高度：`min(720px, calc(100vh - 64px))`。
- 当视口高度低于舒适最小值时，纸张继续收缩到可用高度，不产生视口外溢出。
- 内容 padding 使用 `clamp()` 连续变化，具体 token 在实现阶段以现有间距尺度表达。
- 当内容超过编辑面可用高度时，只滚动 CodeMirror 内容区；纸张和两个操作按钮保持稳定。

## 技术设计

### 顶层 overlay portal

`mountInputOverlay` 创建一个直接挂在 `document.body` 下的顶层聚焦层。该层同时拥有遮罩和纸张容器，从而形成单一、可预测的层叠上下文。

展开时：

1. 记录 `.assistant-input-wrap` 的原父节点和后继节点。
2. 把同一个 `.assistant-input-wrap` 移入聚焦层的纸张容器。
3. 添加放大态类，显示聚焦层并设置可访问语义。
4. 请求 CodeMirror 重新测量并恢复焦点。

收起时：

1. 把 `.assistant-input-wrap` 插回记录的准确位置。
2. 隐藏聚焦层并移除放大态类。
3. 请求 CodeMirror 重新测量并恢复焦点。

移动现有 DOM 节点不会销毁 `EditorView`，因此无需复制或同步编辑状态。顶层 portal 也使聚焦态不再匹配 Floating 的紧凑宽度选择器。

### 尺寸测量

CSS 负责纸张的连续响应式几何。`ResizeObserver` 观察纸张或编辑器宿主的尺寸变化，在实际边界改变时调用 `EditorView.requestMeasure()`。观察器在销毁 overlay 时断开。

### 提交握手

Composer 的 `onSubmit` 从 fire-and-forget 回调改为返回提交结果的异步接口。流程为：

1. Composer 解析并验证有效内容。
2. 助手确保 conversation 存在并调用 sidecar `send`。
3. `send` 返回 request id 后，助手报告提交成功。
4. Composer 清空内容并收起聚焦纸张。

若创建 conversation、更新必要状态或发送请求失败，提交结果为失败。Composer 保留内容和聚焦纸张；错误继续进入助手错误状态，并使用现有共享 toast 提供聚焦层之上的可见反馈。空内容不会触发提交或收起。

### 可访问性

- 聚焦层使用 `role="dialog"`、`aria-modal="true"` 和可读的 `aria-label`，但视觉上不绘制 Dialog chrome。
- 打开时底层应用进入不可交互状态，关闭时恢复。
- 焦点不会进入底层文档；关闭后回到输入器。
- 关闭和发送按钮都有明确的中文 accessible name。
- 动画遵循 `prefers-reduced-motion`。

### 清理与异常恢复

- `destroy()` 在任何状态下都恢复输入器 DOM 位置，移除聚焦层、事件监听和 `ResizeObserver`。
- 如果原父节点在 overlay 存续期间被移除，收起时回退到传入的 dock 宿主；不把输入器遗留在 `body`。
- Floating/Inline 在聚焦期间切换时只改变底层布局，聚焦层保持打开；收起时回到当前助手 dock。

## 测试策略

### Overlay 单元测试

- 展开会创建顶层聚焦层并移动原 host。
- 收起和销毁会把 host 恢复到正确位置。
- 点击遮罩不会收起，`Escape` 和关闭按钮会收起。
- 引用 popover 打开时，第一次 `Escape` 由 popover 消费。
- 展开、连续 resize 和收起均请求 CodeMirror 测量。
- destroy 会断开观察器并移除所有新增 DOM 和监听器。

### Composer 单元测试

- 展开/收起不改变文档、选择和引用模型。
- 成功提交清空并收起。
- 失败提交保留文档并保持展开。
- 空提交不收起，也不在聚焦态打开历史入口。
- Enter、Shift+Enter、IME 和候选键盘行为保持现有测试通过。

### CSS 与 DOM 回归测试

- 聚焦层直接位于 `body`，不受 `#assistant-region` 层叠上下文限制。
- 聚焦态规则不依赖 `.mode-floating` 或 `.mode-inline`。
- Floating 的 `240px` 紧凑规则不匹配 portal 中的 host。
- 放大态 CodeMirror 无边框、无独立背景，并为悬浮按钮留出安全区。
- 小窗口宽高公式不会产生横向或纵向溢出。

### 手工验证

- 分别从 Floating 和 Inline 打开、输入、引用、缩放窗口、关闭和发送。
- 在放大期间切换助手模式并确认纸张不跳变。
- 检查 macOS 与 Windows 的 fixed positioning、viewport 单位、focus ring、IME 和滚动。
- 检查浅色、深色和减少动态效果设置。
