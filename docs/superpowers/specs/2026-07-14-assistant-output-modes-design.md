# AI 输出模式与过程段设计

## 背景

FloatNote 的助手在实时输出期间会把正式文本、思考和工具调用渲染为平级块，但会话恢复只重建用户文本和 AI 正式文本。原始 sidecar 会话已经把 `thinking`、`toolCall`、`toolResult` 和正式文本写入 JSONL；丢失发生在恢复数据转换和前端投影阶段。

本设计解决三个相关问题：

1. 详细模式下，历史思考和工具调用在完成一轮回复、切换会话或重启应用后仍然存在。
2. 助手提供默认简洁模式和可选详细模式，两种模式只改变前端投影，共享同一份完整历史。
3. 详细模式把连续的思考与工具调用组织为紧凑过程段，并显示具体、可读的工具标题。

## 设计原则

- 完整历史只有一个事实源，不因显示模式产生两套数据。
- 简洁和详细是前端展示策略，不改变模型输入、工具执行或持久化内容。
- 正式正文是过程段的唯一分隔符。
- 工具标签说明做了什么，但不暴露工具参数全文、文档正文或工具返回正文。
- 成功状态保持安静；失败状态明确、可定位。
- 前端 reducer 和渲染层继续使用稳定消息 ID、块 ID 与工具 `callId`，避免增量输出闪烁或结果错配。

## 持久化与恢复

### 事实源

`~/.floatnote/chat-history/sessions/<conversation>.jsonl` 继续作为完整会话事实源。Pi `SessionManager` 以追加方式保存用户消息、AI thinking、正式文本、工具调用、工具结果和分支关系。

`~/.floatnote/chat-history/index.json` 继续用于对话列表、标题、作用域、模型、时间和正文摘要。它不承担完整消息恢复，也不复制 thinking 或工具返回正文。

### 恢复投影

sidecar 打开会话时遍历当前活动分支，将原始 session entry 转换为前端可消费的结构化消息：

- 用户消息转换为用户正文。
- AI 内容按原始顺序转换为 thinking、text 和 tool 项。
- `toolCall` 通过 `toolCallId` 与后续 `toolResult` 匹配，恢复 `succeeded` 或 `failed` 状态。
- 工具参数只用于生成安全的显示目标；不把参数对象原样发送给前端。
- 工具结果只用于确定状态和提取经过清理的短错误原因；不把返回正文发送给前端。
- 应用退出时仍没有对应结果的历史工具恢复为 `incomplete`，使用失败视觉但显示“未完成”。

现有原始 JSONL 已包含所需信息，不迁移或重写旧 session 文件。旧会话在下次打开时通过新转换器获得结构化 UI。

## 前端数据模型

一条 assistant 消息保持有序块序列，块分为：

- `text`：正式正文。
- `thinking`：可展开的思考。
- `tool`：不可展开的工具结果行。
- `process_group`：两个及以上连续过程项组成的折叠块。
- `error`：对话级错误。

thinking 与 tool 都是过程项。单独一个过程项保持紧凑单行；第二个连续过程项到达时，将两项合并为 `process_group`。后续 thinking 或任意工具继续追加到同一组，直到正式 `text` 到达。新的过程项出现在正文之后时，新建下一过程段。

thinking 项在过程段展开后仍有自己的二级折叠状态。工具项永远不展开，不显示参数或返回内容。

## 显示模式

### 简洁模式

简洁模式是默认值：

- 只渲染用户消息、AI 正式正文和对话级错误。
- thinking、tool 和 process group 保留在状态中，但不生成可见标签。
- AI 正在 thinking 或执行工具且尚无新正文时，在最后一个可见 AI 输出位置显示闪烁光标。
- 如果本轮尚无正式正文，光标作为助手流中的独立进度标记出现。
- 正式正文开始流式输出后，光标跟随最后一个正文气泡的内容末尾。
- 回复完成或失败后移除光标。

### 详细模式

- 渲染 thinking、tool 和 process group。
- 不显示独立闪烁光标。
- 单个运行中的 thinking/tool，或包含运行项的 process group，使用从左到右的柔和流光表示正在进行。
- 正文流式输出依靠文本本身的增量变化表达进度，不附加光标或流光。
- 完成后的过程段默认折叠；用户可展开查看过程项。

切换模式时只重新投影当前状态，已经载入的消息立即变化，不重新请求 AI，也不改变持久化数据。

## 过程段摘要与状态

过程段标题按状态显示：

- 运行中：`正在处理 · 5 个步骤`
- 全部完成：`处理了 5 个步骤`
- 含失败项：`处理了 5 个步骤 · 1 项失败`
- 含未完成项：`处理了 5 个步骤 · 1 项未完成`

状态视觉规则：

- `running`：过程项或过程段标题显示流光。
- `succeeded`：不显示对勾或“成功”；使用普通弱化文本。
- `failed`：对应工具行使用危险色并追加“失败”；若有安全短错误，显示在同一行或下一行。
- `incomplete`：使用危险色并追加“未完成”。
- process group 标题在包含失败或未完成项时同步显示计数，但其他成功项不增加状态图标。

尊重 `prefers-reduced-motion`：关闭流光和闪烁动画，以静态强调状态替代。

## 工具标题

工具标题由统一的纯函数生成，实时事件与历史恢复共用，避免标签不一致。标题优先使用安全、短小的显示目标：

| 工具 | 标题示例 |
| --- | --- |
| `read_note` | `读取 piece.md` |
| `list_notes` | `列出项目文档` |
| `create_note` | `创建 Ideas.md` |
| `edit_note` | `编辑 piece.md` |
| `write_note` | `编辑 piece.md` |
| `list_tags` | `列出标签` |
| `set_tag` | `给“块首行摘要”设置标签` |
| `tag_create` | `新建标签 行动清单` |
| `tag_update` | `修改标签 行动清单` |
| `tag_delete` | `删除标签 稍后处理` |
| `read_skill` | `读取技能 brainstorming` |
| `web_search` | `搜索网页 FloatNote Tauri` |
| `web_fetch` | `读取网页 example.com` |

系统文件使用产品语义名称，而不是磁盘文件名：

- `_tasks.md` 显示为 `行动清单`，例如 `编辑 行动清单`、`读取 行动清单`。
- `_inbox.md` 显示为 `采集区`，例如 `编辑 采集区`、`读取 采集区`。
- 其他 piece 和 loose document 使用正常文件名。
- 没有显式目标时，根据当前活动文档生成 `读取当前文档` 或 `编辑当前文档`；无法安全确定目标时退回通用工具名称。

显示目标必须截断过长文本、移除换行，并避免把正文型参数（如 `content`、`old_string`、`new_string`）用于标题。

## 设置与跨窗口同步

持久化配置增加 assistant 输出模式字段，取值为 `compact` 或 `detailed`，默认 `compact`。旧配置通过 serde/default 和前端缺省处理自动迁移到简洁模式。

设置页 AI 分类增加“输出显示”控件：

- 简洁：只显示正式内容，使用光标表示隐藏过程仍在进行。
- 详细：显示思考和工具过程，使用过程流光表示运行状态。

保存后向已打开的笔记窗口发送模式变化事件。助手在挂载时读取持久化值，并在运行期间监听该事件；收到后更新投影，无需重新打开会话。

## 错误处理

- 无法解析某个历史内容块时跳过该块并继续恢复同一会话，不能导致整个会话打不开。
- 未知工具使用稳定的原始工具名作为后备标题。
- 找不到工具结果时只标记该工具为未完成，不推断为成功。
- 工具错误摘要限制长度并移除多行正文；详细返回内容只留在原始 JSONL。
- 模式配置值非法时回退到 `compact`。
- 设置保存失败时恢复控件旧值，并保持当前助手投影不变。

## 测试策略

### Sidecar

- 从包含 thinking、text、toolCall 和 toolResult 的真实 session 形状恢复有序结构。
- 匹配并恢复成功、失败、未完成工具状态。
- 工具标题能识别普通 piece、`_tasks.md`/行动清单、`_inbox.md`/采集区、当前文档和网页域名。
- 恢复消息不携带工具返回正文或正文型参数。

### Rust

- 结构化恢复协议的 serde 往返测试。
- 新配置字段默认值、非法值归一化和旧配置兼容测试。
- 跨窗口模式事件只在配置保存成功后发出。

### Frontend reducer 与渲染

- 任意 thinking/tool 连续两项成组，不同工具类型仍在同一组。
- 只有 text 切断过程段；thinking 不切段。
- 单过程项不强制成组。
- tool end 通过 `callId` 更新正确项，包括组内交错完成。
- thinking 在过程段中保持二级展开能力，工具不可展开。
- 简洁模式隐藏过程但保留状态，并正确放置/移除光标。
- 详细模式没有独立光标，流光只出现在运行过程项或过程段。
- 成功静默、失败危险色、未完成恢复状态和 reduced-motion 样式。
- 运行时切换模式立即重新投影已有消息。

### 集成回归

- 完成一轮含多次 thinking 和多种工具的回复后，开始下一轮不会删除上一轮过程段。
- 关闭并重新打开应用/会话后，详细模式仍能看到原有 thinking 与工具记录。
- 同一会话分别在简洁和详细模式打开时，正式正文内容和底层历史完全一致。

## 非目标

- 不在工具标签中展示或保存可展开的参数、返回正文、diff 或工具日志。
- 不改变 permission 气泡的确认流程。
- 不改变模型 thinking level 或要求模型生成更多/更少思考。
- 不为简洁和详细模式分别保存历史。
- 不重写或压缩现有 session JSONL。
